# medi-diff 架构说明（面试速览）

> 手机阅读版 · 按面试提问逻辑组织

---

## 一、一句话定位

基于 Stable Diffusion 1.5 + LoRA 的乳腺钼靶图像生成系统。不是纯算法实验——**数据→训练→推理→质控→API→UI→LLM 顾问**全链路，可独立交付的产品。

---

## 二、四层架构

```
┌────────────────────────────────────────────┐
│  表现层  Gradio UI · FastAPI REST · /docs  │
├────────────────────────────────────────────┤
│  编排层  一键流水线 · 调参状态持久化         │
├────────────────────────────────────────────┤
│  AI引擎  生成(SD+LoRA) · 评估(16维规则)     │
│          · 顾问(多模态LLM)                  │
├────────────────────────────────────────────┤
│  数据层  CBIS_CLEAN_V2 · LoRA权重 · outputs │
└────────────────────────────────────────────┘
```

### 2.1 表现层（apps/）

- **Gradio UI**（981行）：生成单面板、画廊、评估、一键流水线、调参历史共5个Tab。默认密度scattered，全图模式。
- **FastAPI**（468行）：RESTful接口，异步Job队列（threading+UUID），`/health`含GPU/内存信息，请求日志中间件，静态资源挂载。
- **UI和API通过subprocess调脚本**，不直接import——GPU进程隔离，CLI可独立复现。

### 2.2 编排层（scripts/assistant/）

- `run_generate_eval_advise.py`：生成→评估→顾问三阶段串行编排，支持skip开关分段调试。
- `tuning_state.py`：读写`LATEST_NEXT_RUN.json`和`PARAM_HISTORY.json`（最近5轮），支持回滚重跑。
- `ask_advisor.py`：调DeepSeek/GLM/Qwen文本API + Qwen-VL视觉API，输出结构化调参建议。

### 2.3 AI引擎层

**生成（scripts/generation/run_mammo_sd15.py，957行）：**
- 默认全图单次img2img（非patch拼接）
- 源图缩放到长边768，推理后LANCZOS4上采样到长边2048
- 默认参数：strength=0.44, guidance_scale=7.5, num_steps=40, scheduler=dpm
- LoRA权重：mammo_sd15_v6_allMLO/final_lora（r=32，全MLO视图训练）

**评估（scripts/evaluation/review_generated_images.py，2500行）：**
- 16维规则分项（A构图表/B灰度组/C纹理组/D伪影组/E分布组/F解剖组）
- hard_tags一票否决：BANDING/SHAPE_ODD/ARTIFACT_BUBBLES/CONTOUR_FRACTURED等
- soft_reasons：SKIN_LINE_MISSING/GRID_SEAM等只影响分数不否决
- strict vs calibrated双口径防止评估虚高
- BRISQUE参考指标，非hard tag

**源图筛选（filter_source_pool）：**
- 元数据筛选：体位(MLO/CC)、密度(scattered/dense等)
- 宽高比过滤：超过2.2的极端比例丢弃
- 形状质量预筛：Otsu提取乳房区域，圆形度≥0.30，凸缺陷≤0.45，面积占比8%-85%
- 背景伪影检测：背景区亮斑过多→降级为备选池

### 2.4 数据层

- 训练数据：CBIS_CLEAN_V2/metadata_clean.csv（已做mask外标签清零）
- 主线权重：outputs/lora/mammo_sd15_v6_allMLO/final_lora
- 每批次产物：生成图 + source_map.json + run_params.json
- 评估产物：summary.json + review_report.csv

## 三、核心技术决策（必背）

| 决策 | 为什么 |
|------|--------|
| SD1.5+LoRA 而非 SDXL/GAN | SDXL单卡训练太久；GAN不稳定易模式崩溃；SD1.5轻量+LoRA引导性价比最高 |
| 全图推理 而非 Patch拼接 | Patch接缝产生BANDING硬缺陷，后处理治标不治本；768推理+2048上采样平衡显存和质量 |
| 规则评估 而非 分类器 | 医疗场景需要可解释——每个hard tag对应具体视觉缺陷 |
| strict/calibrated双口径 | auto-calibrate会过度放松阈值；hard_tags不受校准影响，始终一票否决 |
| subprocess隔离 UI和脚本 | GPU进程独立，CLI可单独复现，Gradio不拖垮模型内存 |
| GenParams dataclass | 单一真相源，所有参数默认值集中定义，避免界面和命令行不一致 |

## 四、关键难点（STAR故事 · 每个1分钟）

### 难点1：Patch接缝→全图推理

- **问题：** 早期Patch拼接方案在相邻patch边界产生条带伪影（GRID_SEAM/BANDING）
- **排查：** FFT频谱分析确认条带频率特征，对比全图单次推理无此问题
- **结论：** 架构问题非后处理问题——SD在相邻patch边界隐空间信息不一致
- **方案：** 删除patch主线，统一全图单次推理，768推理+LANCZOS4上采样2048
- **结果：** 接缝类缺陷消除，删除约800行patch代码

### 难点2：DICOM标签伪影

- **问题：** 生成图出现"R MLO"文字、菱形修补块——SD学到了训练数据里的烧录标签
- **排查：** 对比inpaint前后→定位是训练数据问题→两头发力
- **方案：** 训练侧mask外清零（CBIS_CLEAN_V2）+ 推理侧label_guard启发式检测擦除 + 源图artifact预筛
- **结果：** 伪影类失败率下降，排查路径可复现

### 难点3：评估虚高

- **问题：** auto-calibrate后pass_rate=100%但肉眼仍有缺陷
- **排查：** 发现校准把BANDING最低分降到0，失去鉴别力
- **方案：** strict（固定阈值+hard_tags不变）和calibrated（参考真实基线调软分）双口径
- **关键：** hard_tags不受校准影响，结构性缺陷始终一票否决

## 五、数据流（一次完整生成评估流程）

```
metadata_clean.csv
      │
      ▼
filter_source_pool()  ──── 体位/密度筛选 + 形状预筛
      │
      ▼
读取源图 → resize长边768
      │
      ▼
SD1.5 + LoRA img2img推理（strength=0.44, steps=40, dpm）
      │
      ▼
LANCZOS4上采样长边2048
      │
      ▼
label_guard后处理（背景标签擦除 + 边缘羽化 + 金属标记修复）
      │
      ▼
输出PNG + source_map.json + run_params.json
      │
      ▼
review_generated_images.py 16维评估
      │
      ▼
summary.json + review_report.csv
      │
      ▼
（可选）LLM顾问读取summary，输出调参建议 → LATEST_NEXT_RUN.json
```

## 六、容错设计

| 场景 | 策略 |
|------|------|
| 源图质量差 | filter_source_pool预筛，圆形度/凸缺陷不达标丢弃 |
| GPU显存不足 | 推理限制768，不直接2048 |
| LoRA权重缺失 | logger.warning继续，可用base model调试 |
| LLM API超时 | 文本60s超时，Qwen-VL每图60s超时 |
| 子进程失败 | 检查returncode，不silent pass |
| 并发写jobs dict | threading.Lock保护 |

## 七、工程规范体现

- **GenParams dataclass**：所有生成参数单一来源
- **source_map.json**：每批次可追溯到源图
- **run_params.json**：每批次参数完整记录，可复现
- **.env管密钥**：代码零硬编码
- **模块级logger**：`logging.getLogger(__name__)`，不用print
- **类型注解**：`X | None`、`Literal`、dataclass
- **向量化优先**：NumPy切片和布尔索引，不写Python逐像素循环

## 八、面试高频追问速答

**Q: 为什么LoRA rank=32？**
A: 试过16和64，32在生成质量和文件大小之间平衡最好。没系统记录对比实验是遗憾。

**Q: strength=0.44怎么确定的？**
A: 实验迭代。0.3以下结构太接近源图（像copy），0.5以上结构变形。0.44保留胸大肌和乳房轮廓。

**Q: 为什么不部署到公网？**
A: 需要NVIDIA GPU推理，至少8GB显存。不适合免费的云部署。

**Q: 评估16维怎么来的？**
A: 从医学物理角度拆解乳腺图像质量要素——解剖完整性、灰度分布、纹理自然度、伪影检测。不是随便凑的维度。

**Q: postprocess为什么归档了？**
A: 频域校正会放大高频噪声为灰色斑点，目视效果更差。功能移除但脚本保留供复现。

**Q: Gradio不是React，能适应吗？**
A: 后端是标准REST+Pydantic模型，和前端框架无关。specForge项目里我用了Next.js+TypeScript，React的组件化和状态管理概念熟悉。前端框架是工具切换，架构思维是一致的。
